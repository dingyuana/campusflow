# 《大模型多智能体系统：智慧校园全栈实战》实训教学大纲

**课程周期：** 10 - 14 天（强化实训，可根据学生基础灵活调整Day6-8的实操时长）
**适用对象：** 计算机、人工智能相关专业大三/大四学生（前置基础：Python基础、简单SQL语句、Git基本使用，无前置基础可增加0.5天前置基础扫盲）
**实训主题：** 基于 LangGraph 的多智能体协作系统开发与云端交付
**核心目标：** 从“技术使用”到“工程落地”，培养学生的**多智能体架构设计能力**、**异构数据融合能力**、**协议对接能力**和**企业级CICD交付能力**，最终完成一个可上线、可交互的智慧校园多智能体问答/办事系统。

## 一、 实训目标（分层拆解，适配教学评价）

### 基础目标（全员掌握，占考核60%）

1. 掌握LangGraph的核心概念（状态机、节点、边、Checkpointer），能实现简单的多智能体顺序/条件编排；
2. 会操作Supabase/Neo4j/Chroma三种异构数据库，能完成基础的数据写入、查询、关联检索；
3. 理解MCP协议的核心思想，能调用现成的MCP Server/API实现工具对接，完成校园导航等基础功能；
4. 掌握Git Flow工作流、GitHub Actions基础配置和Vercel一键部署，能将项目发布至公网预览地址。

### 进阶目标（部分掌握，占考核30%）

1. 能根据业务需求设计多智能体的角色分工和动态路由逻辑，解决“智慧校园”中的实际业务问题（如报到闭环、政策咨询）；
2. 能优化异构数据的检索效率（如RAG语义切分优化、Neo4j多跳查询优化、Supabase索引设计）；
3. 能自定义开发简单的MCP Server，实现自研业务系统与大模型的对接；
4. 能实现系统的基础优化（如断点续传、流式渲染、隐私护栏）。

### 拔高目标（选做，占考核10%）

1. 能针对多智能体系统的瓶颈做性能调优（如节点并行执行、状态缓存、检索结果重排序）；
2. 能完善CICD流水线（如添加单元测试、代码覆盖率检查、环境隔离部署）；
3. 能扩展系统功能（如多轮对话记忆、多用户权限管理、智能体错误重试机制）。

## 二、 实训大纲（按天规划，**优化实操步骤+补充教学要点+增加排错环节**）

### 前置准备（0.5天，可选，针对无前置基础学生）

1. 环境配置：Python3.9+、Anaconda/虚拟环境、Git、Docker（简化数据库部署）、VSCode+插件；
2. 基础扫盲：Python函数/类、简单SQL语句、Git clone/commit/push/branch、FastAPI基础运行；
3. 工具准备：LangSmith账号、GitHub账号、Supabase账号、Neo4j Aura/Chroma Cloud（提供免费额度）。

### 第一阶段：基础设施与异构数据构建 (Day 1-3)

**核心目标：** 搭好项目工程骨架，完成三种异构数据库的基础部署和数据入库，实现“数据层”的搭建。

### Day 1: 项目启动与工程化初探（**补充Docker化部署，降低环境问题**）

- **理论（2h）**
    1. 多智能体系统（MAS）核心设计模式（分工协作、主从编排、动态路由）；
    2. 分布式系统中的状态管理核心问题（数据一致性、状态丢失、多端同步）；
    3. 工程化开发规范：Git Flow工作流（主干/开发/特性/发布分支）、项目目录结构设计（前后端分离、分层架构）。
- **实操（6h，分步骤落地，配沙盒模板）**
    1. 项目初始化：基于模板创建仓库，配置Python虚拟环境/依赖（requirements.txt/pyproject.toml），设计标准项目目录（`api/`/`agents/`/`db/`/`utils/`）；
    2. Git配置：创建dev/feature分支，配置.gitignore，编写Commit规范（Conventional Commits）；
    3. 云端环境初始化：创建Supabase项目，关联GitHub仓库，配置Supabase云数据库（一键创建，免本地部署）；
    4. 基础测试：编写简单的Python脚本，实现本地与Supabase的连接测试，完成第一条数据插入/查询。
- **教学要点**：
  1. **虚拟环境是必选项**：强调每个项目都必须使用虚拟环境（venv），避免依赖冲突；演示如何创建、激活、退出虚拟环境；
  2. 重点解决学生的**环境配置问题**，提供现成的仓库模板和依赖文件，避免因环境问题耽误进度；
  3. 强调项目目录结构的重要性，为后续多模块开发做铺垫。
- **课后作业**：提交Git仓库地址，完成PostgreSQL连接测试截图，推送到feature/day1分支并创建PR。

### Day 2: 知识获取（RAG + 向量库）（**补充语义切分实操+RAG基础评估**）

- **理论（2h）**
    1. RAG的核心流程（加载-切分-向量化-存储-检索-生成）；
    2. 混合检索策略（关键词检索+语义检索）、语义切分原则（按语义完整性、固定长度+重叠窗口）；
    3. Chroma DB的核心特性（轻量级、嵌入式、支持本地/云端部署）、Embedding模型的选择（开源轻量模型如BGE-m3，适配实训）。
- **实操（6h，基于LangChain完成端到端RAG）**
    1. 数据处理：使用LangChain的`PyPDFLoader`/`UnstructuredLoader`处理《校园报到手册》（PDF/Word），完成文本清洗、语义切分（配置切分参数：chunk_size/chunk_overlap）；
    2. 向量化与存储：调用开源Embedding模型生成向量，将文本+向量+元数据存入Chroma（本地/云端）；
    3. 基础检索：实现语义检索、关键词检索、混合检索，对比不同检索方式的结果差异；
    4. RAG基础评估：手动验证召回准确率，调整切分参数和检索策略。
- **教学要点**：重点讲解**语义切分的坑**（如切分后语义断裂、过长/过短切分影响检索），提供现成的文本处理工具类；推荐使用**本地Chroma**降低学生的云端配置成本。
- **课后作业**：完成《校园报到手册》的RAG系统搭建，实现任意问题的检索，提交检索结果截图。

### Day 3: 复杂关系处理（知识图谱 Neo4j）（**补充图建模实操+Text-to-Cypher避坑**）

- **理论（2h）**
    1. 图数据库核心概念（点Node/边Relationship/属性Property），与关系型数据库的区别；
    2. 智慧校园场景的图建模原则（实体抽象、关系定义、属性设计）；
    3. Text-to-Cypher原理（大模型将自然语言转为Cypher语句）、核心问题（语句错误、查询效率低、死循环）。
- **实操（6h，从建模到查询的端到端实现）**
    1. Neo4j环境部署：使用Neo4j Aura（免费云实例）或Docker本地部署，完成连接配置；
    2. 图建模与数据入库：抽象校园核心实体（学生/教师/实验室/课程/院系），定义实体关系（隶属/授课/入驻/选修），编写Cypher脚本导入仿真数据；
    3. 基础查询：实现单跳/多跳查询（如“查询计算机学院张老师的实验室里的学生”）；
    4. Text-to-Cypher实现：基于LangChain搭建Text-to-Cypher系统，添加语句校验和错误重试，避免死循环；
    5. 跨库关联：实现Chroma（非结构化）与Neo4j（结构化关系）的简单混合检索。
- **教学要点**：重点讲解**图建模的合理性**（避免实体冗余、关系混乱），提供智慧校园的标准图模型和Cypher导入脚本；针对Text-to-Cypher，添加**基础的语句过滤**（如禁止DELETE/DROP语句）。
- **课后作业**：完成校园知识图谱的搭建，实现3个以上的多跳查询，提交Neo4j可视化截图和Text-to-Cypher测试结果。

### 第二阶段：数据库深度集成与状态持久化 (Day 4-5)

**核心目标：** 完成业务数据库的设计与开发，实现LangGraph的状态持久化，解决“多智能体系统断点续传”的核心问题，实现“数据层+业务层”的融合。

### Day 4: 业务数据库设计（Supabase）

- **理论（2h）**
    1. 关系型数据库的设计原则（三大范式、反范式优化）；
    2. 智慧校园核心业务表的设计思路（学籍、财务、宿舍等），事务一致性与Supabase RLS策略；
    3. FastAPI核心特性（接口定义、请求/响应模型、路径参数/查询参数、Swagger文档自动生成）。
- **实操（6h，从表设计到接口开发的端到端实现）**
    1. 业务表设计：基于仿真数据，设计`student_records`（学籍）、`payments`（财务）、`dormitory`（宿舍）、`user_preference`（用户偏好）表，添加合理的主键、外键、索引；
    2. 数据入库：编写SQL脚本/Python脚本，将仿真数据导入PostgreSQL业务表；
    3. FastAPI接口开发：分层开发（数据层DAO/业务层Service/接口层API），实现核心业务接口（身份确认、缴费查询、宿舍查询）；
    4. 接口测试：使用FastAPI自带的Swagger UI或Postman完成接口的增删改查测试，保证接口的正确性；
    5. 数据库优化：对慢查询添加索引，优化SQL语句（如避免SELECT *、合理使用JOIN）。
- **教学要点**：强调**数据库设计的工程规范**（命名规范、字段类型选择、索引合理使用），提供分层开发的模板，让学生养成良好的代码习惯；重点讲解**接口的幂等性**（避免重复提交）。
- **课后作业**：完成核心业务表的设计与数据入库，开发并测试所有业务接口，提交Swagger UI接口测试截图。

### Day 5: LangGraph 状态持久化（Checkpointer）

- **理论（2h）**
    1. LangGraph状态机的核心原理（State对象、节点执行、边的跳转）；
    2. 状态持久化的必要性（断点续传、多端同步、日志追溯）；
    3. Checkpointer核心组件（`PostgresSaver`/`MemorySaver`），Thread ID的设计与作用（用户会话唯一标识）。
- **实操（6h，从配置到落地，实现断点续传功能）**
    1. LangGraph基础入门：编写简单的LangGraph状态机（2-3个节点），实现基础的顺序执行；
    2. `PostgresSaver`配置：将LangGraph的Checkpoint存储至PostgreSQL，完成与业务数据库的统一管理；
    3. Thread ID设计：为每个学生会话生成唯一的Thread ID，关联用户信息与状态信息；
    4. 断点续传实现：模拟学生退出/刷新页面场景，通过Thread ID从PostgreSQL加载历史状态，实现进度恢复；
    5. 状态管理优化：添加状态过期机制（如24小时未活跃则清理状态），避免数据库冗余。
- **教学要点**：重点讲解**State对象的设计**（哪些数据需要持久化、哪些数据临时存储），提供现成的`PostgresSaver`配置模板；通过**实际场景模拟**让学生理解断点续传的核心价值。
- **课后作业**：实现基于LangGraph+PostgresSaver的断点续传功能，模拟3次以上的断点恢复场景，提交测试截图。

### 第三阶段：多智能体脑部开发 (Day 6-8)

**核心目标：** 完成多智能体的核心架构设计与开发，实现智能体的角色分工、动态路由、工具集成和MCP协议对接，这是实训的**核心技术阶段**，实现“架构层+工具层”的融合。

### Day 6: Orchestrator 编排器开发（**补充意图识别优化+条件边实操**）

- **理论（2h）**
    1. LangGraph的高级特性：条件边（Conditional Edges）、动态路由、节点的并行/串行执行；
    2. 多智能体编排器（Orchestrator）的核心作用：意图识别、任务拆解、智能体调度、结果聚合；
    3. 意图识别的核心思路（基于大模型的分类、关键词匹配、规则引擎）。
- **实操（6h，从意图识别到动态路由的端到端实现）**
    1. 意图识别节点开发：基于大模型（如开源轻量模型/API）实现意图分类（闲聊/身份确认/缴费查询/宿舍查询/导航/政策咨询），添加规则引擎兜底；
    2. 编排器核心开发：设计LangGraph的核心架构，定义State对象（用户输入/意图/对话历史/执行结果/当前步骤）；
    3. 条件边与动态路由实现：根据意图识别结果，将任务路由至对应的专业智能体，实现从“闲聊”到“业务流程”的平滑切换；
    4. 基础调度测试：模拟不同用户意图，测试编排器的路由准确性，添加错误处理（如未知意图的兜底回复）。
- **教学要点**：重点讲解**State对象的合理设计**（这是LangGraph开发的核心），提供可复用的State模板；针对学生基础，可先提供**规则引擎+简单大模型**的意图识别方案，降低开发难度。
- **课后作业**：完成编排器的开发与测试，实现至少5种意图的准确路由，提交测试用例和运行结果。

### Day 7: 专业 Agent 实战（Policy, Geo, Memory）

- **理论（2h）**
    1. 专业智能体的设计原则：单一职责、高内聚低耦合、可复用、可扩展；
    2. 不同类型智能体的开发思路：RAG Agent（对接向量库）、Memory Agent（对接业务数据库）、Tool Agent（对接外部工具/API）；
    3. 多智能体的结果聚合：将不同智能体的执行结果整理为自然语言，返回给用户。
- **实操（6h，开发3个核心专业智能体，实现与编排器的对接）**
    1. RAG Agent（政策咨询）开发：对接Day2的Chroma向量库，实现校园政策/报到手册的语义检索与生成式回答，完成与编排器的对接；
    2. Memory Agent（用户偏好/信息记忆）开发：对接Day4的PostgreSQL业务表，实现学生身份、缴费状态、宿舍信息的查询与记忆，支持多轮对话中的信息复用；
    3. 智能体解耦设计：将每个智能体封装为独立的函数/类，通过统一的接口与编排器交互，便于后续扩展；
    4. 多智能体协作测试：模拟完整的业务流程（如“我是张三，我的宿舍在哪里？”），测试编排器+专业智能体的协作效果，实现结果聚合。
- **教学要点**：强调**单一职责原则**，避免智能体功能冗余；提供统一的智能体开发模板，让学生只需关注核心业务逻辑，无需重复编写对接代码。
- **课后作业**：完成3个核心专业智能体的开发，实现至少2个完整的业务流程协作，提交测试用例和运行结果。

### Day 8: 工具集成与 MCP 协议

- **理论（2h）**
    1. MCP（Model Context Protocol）协议标准详解：核心设计思想、请求/响应格式、协议优势（解耦大模型与工具、标准化对接、安全可控）；
    2. MCP Server/Client的核心架构：服务端（提供工具接口）、客户端（大模型/智能体调用）、协议适配层；
    3. 外部API集成的核心要点：接口鉴权、请求参数处理、响应结果解析、异常处理（如API调用失败、超时）。
- **实操（6h，从MCP极简实现到第三方API集成，完成工具层对接）**
    1. MCP协议入门：基于FastAPI开发**极简版MCP Server**，封装校园室内导航数据（如教学楼/宿舍的楼层、房间位置）；
    2. MCP Client开发：在LangGraph中实现MCP Client，调用MCP Server的接口，实现室内导航功能；
    3. 第三方API集成：申请高德/百度地图API密钥，实现API鉴权（如AK/SK签名），封装室外路径规划接口；
    4. Tool Agent开发：将MCP接口和地图API封装为Tool Agent，与编排器对接，实现“校园导航”全流程；
    5. 安全优化：对MCP接口添加基础的身份验证，避免未授权调用。
- **教学要点**：针对实训场景，**不要求学生实现完整的MCP协议**，重点掌握“协议标准化对接”的思想；提供现成的MCP Server/Client模板，让学生快速上手；强调**外部API的安全使用**（避免密钥泄露）。
- **课后作业**：完成MCP Server/Client的开发和地图API的集成，实现校园室内+室外导航功能，提交测试截图。

### 第四阶段：全栈交付与云端运维 (Day 9-10)

**核心目标：** 完成前端界面开发与后端流式通信，实现CICD自动化流水线和Vercel企业级部署，将项目从“本地实操”变为“公网上线”，实现“全栈层+运维层”的融合。

### Day 9: 前端交互与流式渲染

- **理论（2h）**
    1. 前后端分离架构的核心思想，Next.js的优势（服务端渲染、路由优化、无缝对接Vercel）；
    2. SSE（服务器发送事件）核心原理：单向通信、流式传输、实时更新，与WebSocket的区别（适配AI对话的流式渲染场景）；
    3. AI对话界面的设计要点：输入框、消息展示、流式打字效果、Agent思考链展示。
- **实操（6h，基于Next.js实现前端界面，完成与后端的SSE流式通信）**
    1. 前端项目初始化：基于Next.js官方模板创建前端项目，配置基础的页面结构（对话页、导航栏、帮助页）；
    2. 后端SSE接口开发：在FastAPI中实现SSE接口，支持AI回答的流式输出，添加Agent思考链的实时推送；
    3. 前后端通信：在Next.js中实现SSE客户端，对接后端SSE接口，实现AI回答的**流式打字效果**；
    4. 思考链展示：添加“Agent思考链”实时展示面板，展示编排器的意图识别结果、智能体调度过程、工具调用记录；
    5. 前端基础优化：添加样式美化（Tailwind CSS）、响应式设计（适配电脑/平板）、加载状态、错误提示。
- **教学要点**：针对非前端专业的学生，**提供现成的Next.js+Tailwind CSS模板**，只需修改接口地址和少量样式，降低前端开发难度；重点讲解**SSE的核心实现**，而非前端样式设计。
- **课后作业**：完成前端界面开发与前后端SSE流式通信，实现完整的AI对话和思考链展示，提交前端运行截图。

### Day 10: CICD 自动化与 Docker 生产部署

- **理论（2h）**
    1. 容器化部署的核心概念：Docker镜像、容器、编排、微服务架构；
    2. 持续集成/持续部署（CICD）的核心原则：自动化、可重复、可追溯、环境一致性；
    3. GitHub Actions的核心组件：工作流（Workflow）、作业（Job）、步骤（Step）、运行器（Runner）；
    4. Docker生产部署的优势：环境隔离、易于扩展、版本控制、快速回滚；
    5. 企业级部署的核心要点：多阶段构建、健康检查、日志收集、监控告警。
- **实操（6h，从GitHub Actions配置到Docker生产环境部署，完成全流程交付）**
    1. Docker基础：编写Dockerfile（多阶段构建），构建API服务和前端服务镜像，测试本地Docker运行；
    2. Docker Compose编排：编写docker-compose.yml，配置服务依赖（ChromaDB、Redis），实现一键启动；
    3. 代码规范与单元测试：添加代码检查工具（flake8/black），编写简单的单元测试（针对核心接口/智能体）；
    4. GitHub Actions配置：编写Workflow文件（.github/workflows/ci-cd.yml），实现**代码提交自动触发**：代码检查→单元测试→Docker镜像构建→推送到GitHub Container Registry；
    5. 生产环境部署：配置生产服务器，使用docker-compose.prod.yml部署到云服务器，配置Nginx反向代理和SSL证书；
    6. 部署验证与运维：访问公网地址，测试全流程功能（报到闭环、断点续传、导航、政策咨询），学习Docker日志查看、容器监控、版本回滚。
- **教学要点**：**提供现成的Dockerfile和docker-compose模板**，学生只需理解原理并少量修改配置；**重点强调敏感信息的管理**（所有密钥/连接信息都通过环境变量配置，禁止硬编码到代码中）；讲解**多阶段构建**优化镜像体积，以及**健康检查**确保服务稳定运行。
- **课后作业**：完成GitHub Actions配置和Docker生产环境部署，提交项目公网地址、CICD流水线运行截图和Docker容器状态截图，确保所有功能在公网环境正常运行。

### 拓展阶段（Day 11-14，可选，针对学有余力的学生）

1. 系统性能优化：实现LangGraph节点的并行执行、Chroma检索结果重排序、PostgreSQL查询缓存，降低端到端响应延迟；
2. 安全防护升级：实现Guardrails护栏（过滤隐私问题、敏感词）、用户权限管理（学生/老师/管理员）、接口限流，防止恶意调用；
3. 功能扩展：添加多轮对话记忆优化、语音输入/输出、校园公告自动更新（定时同步至Chroma）、智能体错误重试机制；
4. 高级CICD：添加代码覆盖率检查、自动化UI测试、镜像构建（Docker）、多环境部署（开发/测试/生产）。
